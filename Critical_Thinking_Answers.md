1) For the authentication flow I went with Firebase Auth because the project required it but also because it made things way simpler than building my own login system. The way I set it up is there's an AuthService class that wraps all the Firebase Auth calls like signIn and signUp and signOut. When a new user registers it also creates a Firestore document in the users collection with their name and email so we have a profile to reference later. The main.dart uses a StreamBuilder on authStateChanges to decide whether to show the login screen or the home screen. That way if someone closes the app and reopens it they stay logged in without having to enter credentials again. The login screen itself toggles between sign in and create account mode using a boolean so we don't need two separate screens for that.

2) Firestore was the backbone of pretty much everything in this app. Every feature writes to and reads from Firestore collections. Tasks go into a tasks collection filtered by uid, rooms go into a rooms collection that tracks occupancy in real time, study groups go into a groups collection with a members array, and chat messages go into a messages collection filtered by groupId. I used StreamBuilder everywhere so the UI updates automatically when data changes on the backend without needing manual refresh. The weekly schedule generator reads pending tasks from Firestore, sorts them by an urgency formula that divides course weight by hours until the due date, then writes time blocks back to a weeklySchedules collection. One thing I had to be careful about was using FieldValue.increment for the room occupancy counter instead of reading and writing the count manually because that would cause race conditions if two users joined a room at the same time.

3) The app uses Firebase Storage as part of the required tech stack. While the current version doesn't have heavy file upload features yet the infrastructure is set up and ready to go. The plan was to let users attach study notes or resources to their groups but we focused on getting the core study planning and collaboration features working first since those were more critical for the demo. The storage dependency is already in pubspec.yaml and Firebase Storage is configured in the project.

4) For FCM we added the firebase_messaging dependency and configured it in the Firebase project. The notification infrastructure is set up so that when reminders need to be sent for upcoming task deadlines or group study sessions FCM can handle the delivery. The main use case we planned for was sending push notifications when a group member starts a shared study session so everyone in the group knows to join. The FCM setup is tied to the same Firebase project that handles auth and Firestore so everything stays under one project.

5) The hardest part of this project was honestly getting the schedule generator to work in a way that made sense. The algorithm takes all your pending tasks and sorts them by urgency which is calculated by dividing course weight by the number of hours until the due date. Higher weight tasks that are due sooner get scheduled first. Then it assigns one hour blocks starting at 9am each day and wraps to the next day after 6pm. It's not perfect but it gives a reasonable weekly plan that students can use as a starting point. If there are no pending tasks it just shows an empty state with a hint to add tasks first. The Firestore reads and writes for the schedule had to be done carefully because generating a new schedule means clearing all old blocks first then writing new ones.

6) Security was something we thought about from the start. Firestore security rules restrict users to only reading and writing their own tasks and schedules by checking request.auth.uid against the document's uid field. For groups we allow read and write only if the user's uid is in the members array. Chat messages are readable by anyone in the group but only writable with a valid senderId. We also added input validation on the client side so empty task titles or invalid effort hours can't be submitted. The Firebase configuration keys are in the app but those aren't really secret since Firestore security rules are what actually protect the data.
